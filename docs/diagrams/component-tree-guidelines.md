# コンポーネントツリー図 作成方針

## 1. 目的

コンポーネントツリー図は **「この画面が何でできているか」を一目で伝える設計成果物** である。
コードを読まずに画面構造を把握できることを第一目標とし、以下の用途で利用する。

| 用途 | 説明 |
|------|------|
| **設計レビュー** | 実装前に構造を合意する。過不足・重複に気付きやすい |
| **テスト設計** | どのコンポーネントをモックするか、何を検証するかを特定する |
| **オンボーディング** | 新メンバーがコードを読む前に全体像を把握する |
| **リファクタリング判断** | コンポーネントが肥大化していないかを視覚的に確認する |

---

## 2. 記述粒度の方針

### 2-1. 記載するもの

| 対象 | 記載するか | 理由 |
|------|-----------|------|
| **自プロジェクトのコンポーネント** | ✅ 必ず記載 | 設計の中心。変更対象になる |
| **Ionic 組み込みコンポーネント（IonList, IonItem 等）** | ✅ 記載する | UIの構造を決定するため省略すると木が曖昧になる |
| **v-if / v-else-if の分岐** | ✅ 記載する | 条件分岐はテスト観点に直結する |
| **slot の受け渡し内容** | ✅ 記載する | 親子間のインターフェースが分かる |
| **イベントハンドラ（@click 等）** | ✅ ノードに注記する | テストの「操作トリガー」特定に使う |
| **v-for の繰り返し** | ✅ `(×n)` と表記 | 動的リストであることを示す |
| **CSSクラス・スタイル** | ❌ 原則省略 | 構造ではなく装飾情報 |
| **プロパティの全一覧** | ❌ 原則省略 | 重要なものだけ注記（:disabled 等） |
| **HTML ネイティブ要素（div, p 等）** | ❌ 省略 | Ionic コンポーネントで置換されている前提 |

### 2-2. 深さの基準

```
ページ（View）
  └── レイアウト（AppLayout 等）
        └── Ionic コンテナ（IonContent, IonList 等）
              └── Ionic 入力・表示コンポーネント（IonItem, IonInput 等）
                    └── ✋ ここで止める（HTML要素まで降りない）
```

- **自作コンポーネントは何階層でも記載**
- **Ionic コンポーネントはその直接の子までを上限とする**
- IonSelectOption のように親と一体で意味をなすものは記載する

### 2-3. 条件分岐の表記

`v-if / v-else-if / v-else` は独立したブランチとして記載し、条件式を明示する。

```
[v-if isLoading]
  └── IonSpinner
[v-else-if isError]
  └── IonText : "読み込みに失敗しました"
[v-else] IonList
  └── IonItem (×n)
```

条件分岐が多い場合は図が縦長になりすぎるため、
**3分岐を超えたらシーケンス図の `alt` ブロックに移管する** ことを検討する。

## 3. ツールと記法

### 3-1. PlantUML WBS（`@startwbs`）を使う理由

- 階層ツリーを短いテキストで表現できる
- Git 管理できる（バイナリではなくテキスト）
- VS Code + PlantUML 拡張でリアルタイムプレビュー可能
- Markdown への埋め込みも可能（Mermaid 代替）

### 3-2. ノードの書き方

```plantuml
* ページ名\n<ファイルパス>          ← ルート（ページ）
** コンポーネント名\n[付加情報]     ← 自作コンポーネント
*** IonXxx : "表示文字列"          ← Ionic コンポーネント
**** [v-if 条件]                   ← 条件分岐
***** 子コンポーネント
```

`\n` で改行して補足情報を添える。`:` の後ろに静的テキストや v-model の変数名を書く。

### 3-3. ファイル命名規則

```
comp-{ページ名またはコンポーネント名}-{補足}.puml
例:
  comp-item-list-page.puml
  comp-item-detail-page.puml
```

## 4. テスト設計への活用

### 4-1. テスト対象の特定

コンポーネントツリーを見て **「この単位でテストを書くか」** を判断する。

```
ItemListPage               ← ページ単位のテスト（統合テスト）
  └── AppLayout            ← レイアウト単位のテスト（スナップショット等）
        └── IonList        ← モックアウト対象（Ionic は実装内部まで検証しない）
              └── IonItem  ← モックアウト対象
```

**自作コンポーネント境界** でテストを分割するのが基本方針。

### 4-2. モックの判断基準

| コンポーネント種別 | テスト方針 |
|-------------------|-----------|
| **自作ページ・レイアウト** | 実コンポーネントをマウントしてテスト |
| **Ionic コンポーネント** | `@vue/test-utils` でスタブ化（`global.stubs`）。Ionic の内部動作はテストしない |
| **外部 API（customFetch）** | MSW でインターセプト。実際の HTTP を模倣する |
| **Vue Router** | `createRouter` で実際のルーターを使うか、`RouterLinkStub` でスタブ化 |
| **Vue Query** | `VueQueryPlugin` を実際にインストールして使う |

### 4-3. テストシナリオとツリーの対応

ツリーの **分岐（v-if）ごとにテストケースを1つ** 用意するのが最低ライン。

```
[v-if isLoading]        → テスト: "ローディング中は IonSpinner が表示される"
[v-else-if isError]     → テスト: "エラー時はエラーメッセージが表示される"
[v-else] IonList        → テスト: "データ取得後はアイテム一覧が表示される"
                        → テスト: "アイテムが空の場合は「項目がありません」と表示される"
```

ツリーに `@click` が記載されているノードは **「クリックしたときに期待動作が起きるか」** のテストが必要。

```
IonButton : "追加" @click="openCreate"   → テスト: "追加ボタンをクリックするとモーダルが開く"
IonButton : "保存" @click="submitCreate" → テスト: "保存ボタンをクリックすると mutation が呼ばれる"
```

## 5. 更新のタイミング

| イベント | 対応 |
|----------|------|
| 新しいコンポーネントを追加した | 該当ページの puml を更新 |
| コンポーネントを削除・移動した | 該当ページの puml を更新 |
| v-if 条件を追加・変更した | 該当ページの puml を更新 |
| リファクタリングで親子関係が変わった | puml を更新し、構造が適切かレビューする |
| Ionic コンポーネントの大幅な入れ替え | puml を更新 |

**コードと図の乖離が最大のリスク。** 実装と同じ PR で図も更新することを原則とする。

---

## 6. やらないこと（スコープ外）

- **Storybook との重複管理は避ける**
  Storybook があればコンポーネントの見た目はそちらで確認できる。
  ツリー図はあくまで「構造の把握」に特化する。

- **全コンポーネントの網羅的な図は作らない**
  汎用コンポーネント（AppLayout 等）は各ページのツリー内に展開して記載する。
  単独で独立した図は作らない（重複が増えてメンテが困難になる）。

- **自動生成は目的を達成しない**
  AST からの自動生成ツールを使うと情報過多になる。
  「何を伝えたいか」を意識して手書きすることで設計の意図が残る。

## 7. サンプル図の構成（このプロジェクト）

```
docs/diagrams/
  comp-item-list-page.puml    : ItemListPage（モーダル・一覧・分岐含む）
  comp-item-detail-page.puml  : ItemDetailPage（404・編集フォーム含む）
```

各図は対応するシーケンス図と合わせて読むと、
**「どのコンポーネントが何をトリガーにどのAPIを呼ぶか」** が把握できる。

```
comp-item-list-page.puml     →  seq-item-list-operations.puml
comp-item-detail-page.puml   →  seq-item-detail-operations.puml
```
