# 項目定義書 作成方針

## 1. 目的

項目定義書は **「この画面にどんな操作・表示・入力があるか」を仕様として記述する成果物** である。
コンポーネントツリー図が「構造」を表すのに対し、項目定義書は **「各要素の意味・制約・ふるまい」** を表す。

| 成果物 | 問いへの答え |
|--------|------------|
| コンポーネントツリー図 | この画面は何でできているか？ |
| **項目定義書** | 各要素はどんなルールで動くか？ |
| シーケンス図 | 操作したときに何が起きるか？ |

3種類を組み合わせることで、実装・レビュー・テストのいずれの観点でも画面仕様を把握できる。

---

## 2. 記述対象の選定基準

### 記載するもの

| 対象 | 記載するか | 理由 |
|------|-----------|------|
| **入力フィールド**（IonInput, IonTextarea, IonSelect） | ✅ | 必須/任意・型・バリデーションが仕様の核 |
| **操作ボタン**（IonButton, IonBackButton） | ✅ | 押下時のふるまい・制約が仕様になる |
| **ナビゲーション要素** | ✅ | 遷移先・履歴への影響が仕様になる |
| **状態表示**（スピナー・エラー・空状態） | ✅ | 表示条件がテスト観点に直結する |
| **選択肢**（IonSelectOption） | ✅ | 選択値の制約・ラベルが仕様になる |
| **表示専用テキスト**（ページタイトル・固定ラベル） | △ | ヘッダーなど文言確認が必要な場合のみ |
| **装飾・レイアウト**（IonContent, IonList） | ❌ | 構造はコンポーネントツリー図で表現済み |
| **アイコン**（IonIcon） | ❌ | 種類を注記する程度で十分 |

### 記載しないもの

- CSS クラス・スタイル（ion-padding 等）
- コンポーネントの内部実装（Ionic が提供するアニメーション等）
- API の詳細なレスポンス構造（別途 OpenAPI 仕様書で管理）

## 3. 項目定義書の構成

### ファイル構成

```
docs/diagrams/
  item-def-{ページ名}.md    ← 各ページの項目定義書
  item-definition-guidelines.md  ← この方針書
```

### ファイル内の構成

```markdown
# 項目定義書 — {ページ名}

ページ情報（ルート・ソース・データ取得API）

## {エリア名}（ヘッダー / コンテンツ / モーダル 等）

| No. | 項目ID | 表示ラベル | コンポーネント | 種別 | 必須 | 型 | デフォルト | 制約・バリデーション | 操作時の挙動 | 対応API |

## フォーム初期値の設定ルール（入力フォームがある場合）

## 状態遷移まとめ

## バリデーション一覧

## テスト観点（この定義書から導出）
```

---

## 4. 各列の定義

| 列名 | 記述内容 | 例 |
|------|---------|-----|
| **No.** | エリア頭文字 + 連番。ヘッダー=H、ローディング=L、エラー=E、コンテンツ=C、モーダル=M | `C-1`, `M-3` |
| **項目ID** | ケバブケースの識別子。テストコードで `findByTestId` 等に使える | `form-title`, `save-btn` |
| **表示ラベル** | 実際に画面に表示される文字列。アイコンのみの場合は「（アイコンのみ）」 | `タイトル`, `保存` |
| **コンポーネント** | Ionic コンポーネント名（または自作コンポーネント名） | `IonInput`, `IonButton` |
| **種別** | 下記「種別の定義」参照 | `入力`, `操作` |
| **必須** | ✅=必須、❌=任意、—=入力なし | `✅` |
| **型** | 入力値の型または選択肢の型 | `string`, `todo \| doing \| done` |
| **デフォルト値** | 初期値。APIから設定される場合はその旨を記載 | `'todo'`, `item.title` |
| **制約・バリデーション** | フロント側の制約（サーバーサイドバリデーションは別管理） | `空文字不可` |
| **操作時の挙動** | ボタン押下・選択時に何が起きるか | `mutation.mutate()` |
| **対応API** | どのエンドポイントを呼ぶか | `POST /api/items` |

### 種別の定義

| 種別 | 意味 | 代表コンポーネント |
|------|------|----------------|
| **入力** | ユーザーが値を入力する | IonInput, IonTextarea, IonSelect, IonCheckbox |
| **操作** | タップで処理を実行する | IonButton |
| **ナビゲーション** | タップで画面遷移する | IonItem（button）, IonBackButton |
| **状態表示** | UIの状態を表示する（ローディング・エラー・空状態） | IonSpinner, IonText |
| **表示** | データを表示するだけ（操作なし） | IonLabel, IonTitle |
| **表示コンテナ** | 子要素のグルーピング（内容物が重要） | IonList, IonContent |
| **選択肢** | 親の選択UIの一部 | IonSelectOption |

## 5. テスト設計への活用

### 5-1. テストケースの導出ルール

項目定義書の各行が **テストケースの入力** になる。

| 列 | テストへの変換 |
|-----|-------------|
| 種別=**状態表示** の行 | `表示条件が true のときコンポーネントが存在する` |
| 種別=**入力** かつ 必須=✅ | `空のとき関連ボタンが disabled になる` |
| 種別=**操作** の行 | `クリックしたとき「操作時の挙動」が実行される` |
| 種別=**ナビゲーション** の行 | `クリックしたとき遷移先へ移動する` |
| **デフォルト値** の列 | `初期表示時にその値が設定されている` |

### 5-2. テストケース密度の目安

```
✅ 必須テスト（仕様の根幹）
  ・状態表示系: 表示条件ごとに1ケース
  ・必須入力: 空入力でのUI制御
  ・操作ボタン: 成功時のふるまい

△ 推奨テスト（品質向上）
  ・操作ボタン: API エラー時のふるまい
  ・初期値の検証
  ・選択肢の数・内容確認

❌ 不要なテスト（コストに見合わない）
  ・Ionic コンポーネント自体の動作（Ionic がテスト済み）
  ・CSS・スタイルの検証
  ・ラベル文字列の一致確認（変更頻度が高く壊れやすい）
```

### 5-3. 項目IDのテストコードへの活用

項目IDをコンポーネントの `data-testid` に付与することで、テストから意味のある形でDOM要素を特定できる。

```vue
<!-- 実装例 -->
<ion-input data-testid="form-title" v-model="form.title" />
<ion-button data-testid="save-btn" :disabled="!form.title">保存</ion-button>
```

```ts
// テスト例
const titleInput = wrapper.find('[data-testid="form-title"]')
const saveBtn = wrapper.find('[data-testid="save-btn"]')
expect(saveBtn.attributes('disabled')).toBeDefined()
```

> **注意:** `data-testid` は開発・テスト専用。本番ビルドで除去したい場合は Vite プラグインで対応する。

## 6. 更新のタイミング

| イベント | 対応 |
|----------|------|
| 新しい入力フィールドを追加した | 該当ページの定義書に行を追加 |
| バリデーションルールを変更した | 制約・バリデーション列を更新 |
| ボタンの挙動を変更した | 操作時の挙動列・テスト観点を更新 |
| 状態の分岐（v-if）を追加した | 状態別表示セクションに追記 |
| API エンドポイントが変わった | 対応API列を更新 |

**定義書とコードの乖離を防ぐため、実装変更と同じ PR で定義書も更新すること。**

---

## 7. コンポーネントツリー図との使い分け

| 知りたいこと | 参照する成果物 |
|------------|-------------|
| 画面の構成要素と階層関係 | コンポーネントツリー図（`comp-*.puml`） |
| 各要素の入力ルール・制約・ふるまい | **項目定義書（`item-def-*.md`）** |
| ユーザー操作でどのAPIが呼ばれるか | シーケンス図（`seq-*.puml`） |
| テストで何を検証すべきか | **項目定義書のテスト観点セクション** |

---

## 8. やらないこと

- **サーバーサイドバリデーションの記載** → API 仕様書（OpenAPI）で管理する
- **デザイントークン・色・余白の記載** → CSS / Figma で管理する  
- **全プロジェクトの全コンポーネントの網羅** → ページ単位で必要なものから作成し、汎用レイアウト（AppLayout）は各ページの定義内に吸収する
- **自動生成** → UIの意図・制約は手書きで残すことで設計の判断が記録される
